# 固有値ページ更新機構の研究方針メモ

## 1. 背景と目的
- 対象サイトは Next.js ベースの数学教育コンテンツ。固有値問題の主要ページは `app/linear_algebra/contents/eigen_solve.tsx` と `app/linear_algebra/contents/eigen_property.tsx`。
- 既存サイトには RAG チャット（`app/api/rag/route.ts` と `app/components/Rag.tsx`）およびコメント経由のコンテンツ更新機構が存在するが、本研究ではそれらを流用せず、新規に「ページ内容を動的に書き換える更新機構」を 2 系統（汎用版 / 専門版）で実装し、比較する。
- 目的は、LLM 生成コメントを入力としてページ説明文・補足・強調点を自動更新し、知識依存度の異なる 2 方式の有効性とコストを評価すること。

## 2. 既存機構の整理（参考のみ）
- コメント投稿/表示
  - 投稿: `app/components/GoogleForm.tsx` から Google Form へ送信。`entry.2025028027` に `app/<topic>/<slug>` 形式のパスを埋め込む。
  - 表示: `app/components/Googlecomment.tsx` が d3 でスプレッドシート CSV を取得し、`display === "1"` かつ `Path` が現在パスに一致する行のみを DOM に描画。
- コメント→修正指示生成
  - `scripts/suggest_improvements.py`: pandas でスプレッドシートを read_html。`Flag == "1"` かつ直近 3 日のコメントを `Path` 単位でまとめ、GPT-4o-mini に要約と修正指示を依頼し、`suggest/<YYYYMMDD>/<path>/xxx.txt` に保存。
- 指示の自動適用
  - `scripts/apply_modifications.py`: 最新の日付ディレクトリを探索し、対応する `app/<dir>/contents/<file>.tsx` を GPT（o3-mini）で全置換。スタイルは変更しない制約。差分表示→手動確認後に書き込み。`metadata.ts` の `lastUpdated` を正規表現で更新。
- 既存 RAG チャット
  - `/api/rag` で動的埋め込み生成＋ Gemini 回答。UI は `app/components/Rag.tsx`。本研究の更新機構とは完全に切り離す。

## 3. 研究で新規に作る 2 方式
- 共通方針
  - 既存コメント表示・既存 RAG チャットとは別ラインとして実装。既存ページ本体は上書きせず、「更新結果の差し替え」用データを別ストレージに置く。
  - 入力コメントは LLM で自動生成したセットを使用し、同一セットを両方式に投入して比較。
  - 更新結果は「挿入・置換のパッチ（セクション単位）」として記録し、復元可能にする。

### 3.1 推奨ディレクトリ/モジュール構成（案）
```
experiments/eigen_update/
  comment_sets/
    seed_prompts.yaml           # カテゴリ別プロンプト種
    generated/2025-XX/comments.json
  knowledge/
    curated_refs.md             # 専門版用の参照リスト（URL/書誌/要約）
    scraped/                    # クローラ取得物 or 手動要約テキスト
  engines/
    schema.ts                   # 入出力・パッチの型定義
    generic_update.ts           # 汎用版: ルール/テンプレ生成
    expert_update.ts            # 専門版: RAG 取得 + 生成
    postprocess.ts              # 数式・強調整形などの共通後処理
  pipeline/
    generate_comments.ts        # LLM でコメントセットを生成・保存
    run_generic.ts              # 汎用版のバッチ適用
    run_expert.ts               # 専門版のバッチ適用
  outputs/
    generic/<timestamp>/eigen_solve.json
    expert/<timestamp>/eigen_solve.json
app/experiments/eigen_update/
  Preview.tsx                   # 2 方式の結果を並列表示する UI（本番ページとは別ルート）
  apply/[mode]/route.ts         # パッチ適用用 API（手動承認付き）
```
- 既存 `scripts/` との分離を明確にし、バッチ実行は `npm run` で呼べるようにする（例: `npm run exp:eigen:generic`）。

### 3.2 汎用更新版（専門知識なし）
- 入力: コメント本文 + 元ページの静的テキスト抽出（既存 `extractTextFromTsx` のような簡易パーサを流用）。
- 生成方針:
  - 文章リライト、見出し下の「要点箇条書き」、誤解パターンへのテンプレ回答を挿入。
  - 固定テンプレ（例: 定義の再掲、計算ステップの手順提示、図示アイデア）をコメントカテゴリごとに差し込む。
  - 外部検索・外部コーパスは使わない。生成モデルにはサイト内コンテンツのみを渡す。
- 出力: パッチスキーマ `Patch = {targetSectionId, operation: insert|replace, content, rationale}` を JSON 保存。DOM 変更は別 UI で確認してから手動適用。

### 3.3 専門更新版（外部知識あり）
- 参照コーパス:
  - `knowledge/` に数学的背景の要約テキストを用意。必要に応じて arXiv/Wikipedia などから取得し、`scraped/` にプレーンテキスト化。
  - 既存 RAG チャットとは別に軽量ベクトルストア（例: ローカル Faiss かシリアライズした JSON）を `engines/expert_update.ts` 内で作成。
- 生成方針:
  - コメント → 意図分類 → 関連節をページ内 + 外部知識から抜粋してプロンプト化。
  - 例示追加、厳密性補足、定理の前提条件・反例の提示を優先。
  - 出典リンク（書誌 or URL）と「どの知識を使ったか」のメタデータを併記し、パッチ JSON に埋め込む。
- 出力: 汎用版と同じパッチスキーマに加え、`citations` と `retrieval_log` を付与。

### 3.4 DOM/レンダリング戦略（比較のための安全策）
- 本番ページは変更せず、`Preview.tsx` で「元コンテンツ + パッチ適用後」を左右に表示。セクション単位で差分を可視化し、アンカーを揃える。
- 承認後に適用する場合も、TSX を直接上書きせず、`outputs/<mode>/<timestamp>.json` を元にコード生成 or MDX 挿入を行い、git で差分管理。
- これにより 2 方式の挙動を同一 DOM 上で比較でき、復元も容易。

## 4. LLM コメント生成の設計
- カテゴリ（例示）: 初学者の素朴な質問 / 誤解を含む質問 / 定義確認 / 証明・性質の深掘り / 具体例・直感要求 / 計算手順の詰まり / 専門的な突っ込み（スペクトル定理、対角化条件など）。
- 生成プロンプト案:
  - 入力: 対象ページの要約 + 想定読者レベル（初学者/中級/上級） + ターゲットカテゴリ + 難易度スカラー（1–5）。
  - 制約: 200–400 字、固有値/固有ベクトルに関連、記号は TeX 記法可、引用や外部参照は禁止。
  - 出力形式: `{category, level, comment, expected_gap}` を JSON Lines で保存し、`comment_sets/generated/<date>/comments.json` に格納。
- 難易度統制: 誤概念を含める割合、計算ステップの欠落数、用語の曖昧さをパラメータ化。

## 5. 比較実験設計
- 手順
  1. コメント生成スクリプトで固定シードのコメント集合を作成。
  2. 汎用版・専門版それぞれに同一集合を投入し、`outputs/` にパッチとログを保存。
  3. `Preview.tsx` で差分を確認し、必要に応じて人手で「妥当/過剰/不足」をラベル付け。
  4. 主要ページ（eigen_solve, eigen_property）で同じ手順を繰り返し、比較表を作成。
- 評価観点（例）
  - 説明の正確性・誤り訂正率
  - 学習者理解の促進度（手順の透明性、直感的説明の有無）
  - ページ内情報との整合性（notation、前後節との一貫性）
  - 更新の一貫性（同カテゴリコメント間の反応の揺れ）
  - DOM 破壊リスク（構造変更の有無、スタイルへの影響）
  - 処理コスト（生成時間、外部リクエスト数、トークン使用量）
- 記録すべきログ
  - 入力コメント、使用したプロンプト、取得コンテキスト（専門版）、生成出力、適用パッチ、diff スナップショット、推論時間。

## 6. 今後の進め方（短期タスク）
1. `experiments/eigen_update/` 配下のスケルトン生成とスキーマ定義（コメント JSON / パッチ JSON）。
2. コメント生成スクリプトの雛形を作成し、カテゴリごとのサンプル 5 件ずつを生成。
3. 汎用版エンジン: 固定テンプレの挿入ロジック（見出し下への要約挿入、例示ブロック）を実装。
4. 専門版エンジン: `knowledge/` に最小コーパスを準備し、簡易ベクトル検索 + citation 付き生成のプロトタイプを作成。
5. `Preview.tsx` で 2 方式の結果を可視化し、評価観点に沿ったチェックリストを作成。
6. 置換型パッチ（replace）で本文を書き換える実験系を追加し、挿入型との比較を行う。

## 7. 現状の比較結果（暫定）
対象: `experiments/eigen_update/outputs/generic/latest.json` と `experiments/eigen_update/outputs/expert/latest.json` の比較

### 7.1 汎用版（テンプレベース）
- **強み**: 出力が安定し、必ず補足が入る。定義や計算手順の補助に強い。
- **弱み**: 専門的な背景説明や反例提示が難しい。出典が無いので信頼性の裏付けが弱い。
- **適用例**: 初学者向けの「定義の再説明」「計算手順のチェックリスト」に向く。

### 7.2 専門版（DB参照 + ベクトル検索 + Gemini）
- **強み**: 参照DBに基づいた補足が可能。専門的な語（直交対角化、重複度など）に対応できる。
- **弱み**: API 失敗時の品質ブレがあり、要約の長さや文体が揺れやすい。
- **適用例**: 「実対称行列の固有値が実数である理由」「対角化条件」など専門性の高い説明に向く。

### 7.3 改善ポイント（現状観測）
- **検索精度**: 固有値関連キーワードでフィルタし、関連外の混入を抑制。
- **生成安定性**: Gemini 503 時の再試行を導入し、フォールバック時も短文化する。
- **文体整合性**: 400字程度でトリムし、本文に近い長さへ抑制。

### 7.4 研究観点での含意
- 汎用版は「安定性」と「実装コストの低さ」で優位。
- 専門版は「説明の深さ」と「専門性」で優位だが、運用コストと品質揺れが課題。
- 研究としては「品質安定性 vs 専門性」のトレードオフが明確に観測できる構図になっている。

### 8.4 多面的評価（定性・定量）
対象: `experiments/eigen_update/outputs/generic/latest.json` と `experiments/eigen_update/outputs/expert/latest.json`

#### 8.4.1 定性評価（視点別）
- **学習者視点**: 汎用版は短く読みやすく、誤解ポイントに絞った補足で学習導線が明確。専門版は説明が厚いが長文で途中省略があり、読解負荷が高い。
- **編集者視点**: 汎用版はそのまま挿入しやすい。専門版は要約・構成整理・出典整備が必須で編集コストが高い。
- **研究/実験視点**: 専門版は「深さ」を出せる一方、品質が安定しない（省略・未完の兆候）が観測できる。汎用版は安定性が高く再現性がある。
- **運用視点**: 専門版は外部ソース参照の体制が必要で運用コストが増える。汎用版は軽量で継続運用向き。
- **品質保証視点**: 専門版は「【出典】」が空である点がリスク。汎用版も `citations` が空でエビデンス不足だが、断定の強さが相対的に低い。

- 専門版を採用するなら、長文化の抑制・省略排除・出典充実の3点が必須タスク。

#### 8.4.6 読み手別の適合性
- **初学者**: 汎用版が適合。短文で「誤解ポイント」へ誘導できる。専門版は説明が冗長で迷子になりやすい。
- **復習目的の学習者**: 専門版の方が背景説明があり補助になるが、未完の箇所で理解が途切れる。
- **講師/教材制作者**: 汎用版は短く差し込み可能。専門版はリライト前提の素材として価値がある。

#### 8.4.7 既存本文との整合性
- **トーン一致**: 汎用版は既存の簡潔な説明調と馴染む。専門版は「補足」色が強く、本文より濃いトーンで浮きやすい。
- **長さのバランス**: 専門版は本文量を上回る恐れがあり、ページ内の比率設計が必要。
- **用語レベル**: 専門版はレイリー商・直交基底など専門語が多く、導入順との整合を要確認。

#### 8.4.8 誤解リスク/欠落リスク
- **誤解リスク**: 専門版の長文は焦点が散り、主張がぼける可能性。汎用版は簡潔だが説明不足で誤解が残るリスク。
- **欠落リスク**: 汎用版は「なぜそうなるか」の根拠が薄い。専門版は根拠を示すが、出典が空で検証性が不足。

#### 8.4.9 表現品質の観点
- **冗長性**: 専門版は冗長になりやすく、重複表現が増えがち。汎用版は冗長性が低い。
- **未完の兆候**: 専門版は省略記号が全パッチに含まれ、完成度が低い。汎用版は未完の兆候がない。
- **構造明確性**: 専門版は箇条書きが多く構造化されるが、情報量が多く読解負荷が上がる。

#### 8.4.10 研究・運用上の示唆
- **品質管理**: 専門版は出典埋めとトリムが必須工程として追加されるため、生成後のレビュー工数が前提になる。
- **安定運用**: 汎用版は失敗時のリカバリが容易で、定期更新の運用に向く。
- **評価設計**: 「深さ vs 安定性」をトレードオフとして明示し、用途別に採用基準を分けるのが実務的。
